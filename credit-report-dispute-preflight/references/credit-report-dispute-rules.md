# Credit Report Dispute Rules

Use these rules as a local evidence checklist, not as legal advice, credit-repair advice, or a guarantee of deletion. Mark current-law or bureau-specific conclusions as "verify before send" when the report source, item identity, dispute history, or current guidance is uncertain.

## Core Decision Axes

- **Item identity**: Require bureau, account or collection name, furnisher, masked account number if available, report date, and the exact field being disputed.
- **Error theory**: Use one narrow, evidence-backed theory per item: identity theft, not mine, paid/settled, wrong balance/status, duplicate, obsolete/stale, mixed file, personal information, or investigation response problem.
- **Bureau vs furnisher routing**: For furnished account data, prepare both a bureau dispute and a furnisher dispute when the furnisher controls the reported data. For personal information and bureau processing issues, bureau routing may be primary.
- **Identity theft**: Fraudulent-account packets usually need an FTC IdentityTheft.gov report, police report, or equivalent sworn report plus the highlighted report page and identity documents with sensitive numbers redacted.
- **Repeat disputes**: A repeated dispute that previously came back verified needs new evidence, a narrower disputed field, or a different routing path. Generic re-disputes are easier to reject as unsupported or duplicative.
- **Reinserted items**: If an item was deleted and reappeared, attach prior deletion or investigation-response proof and identify the bureau/date where reinsertion occurred.
- **Deadline tracking**: Track report pull date, dispute sent date, response date, and any lending/rental/job deadline. Do not promise a bureau response time; use deadlines as follow-up triggers.
- **Packet hygiene**: Use concise factual language: what is wrong, why it is wrong, what evidence proves it, and what correction is requested. Avoid threats, credit-repair myths, or broad "delete everything" demands.

## Evidence Checklist

| Scenario | Minimum useful evidence |
|---|---|
| Identity theft or fraud account | Highlighted report page, FTC/police identity-theft report, dispute letter, identity proof with sensitive fields redacted, any creditor fraud response |
| Not mine or mixed file | Highlighted report page, proof of identity/address, account mismatch facts, records showing different person/entity when available |
| Paid, settled, or wrong balance | Highlighted report page, payoff or settlement letter, payment receipt, account statement, creditor or collector response |
| Wrong status or late payment | Highlighted report page, billing statements, payment history, servicer letter, autopay or bank proof if relevant |
| Duplicate tradeline | Highlighted pages for both entries, account numbers or dates showing duplication, furnisher names, and which item should remain |
| Obsolete or stale collection | Highlighted report page, date of first delinquency or account history evidence, collector/furnisher letter when available |
| Reinserted item | Prior deletion or investigation response, current report page, date item reappeared, and bureau-specific evidence |

## Common Failure Modes

- Dispute says "not mine" but attaches no identity-theft report or account mismatch evidence.
- Packet uses a Credit Karma or monitoring-app screenshot but no official bureau report page.
- Same generic letter is sent to all bureaus without bureau-specific item names and fields.
- User re-disputes an item that was verified, with no new evidence or narrower correction request.
- The disputed item is accurate but unfavorable; the correct output is "do not dispute without a factual error."
- Sensitive files include full SSNs, full ID scans, bank credentials, private keys, tokens, or unrelated third-party records.
- User asks the agent to submit a live bureau/CFPB/creditor portal action instead of preparing a local packet for owner review.
